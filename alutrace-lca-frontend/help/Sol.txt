🔹 Aluminium Industry LCA System: AluTrace


---

1. PROJECT SCOPE & STRATEGY 🗓

Goal: Build an end-to-end Life Cycle Assessment (LCA) tool for aluminium industries to measure, monitor, and reduce environmental impact across the full supply chain (mining → recycling).

Scope:

Covers all stages: Mining, Refining, Smelting, Finishing, Transport, Recycling.

Measures CO₂ emissions, energy intensity, water footprint, and waste generation.

Provides hotspot analysis + what-if scenario planning.


Technology Stack:

Backend: Python + FastAPI (calculation engine, APIs)

Data Processing: Pandas, NumPy (normalization, math ops)

Database: PostgreSQL (raw data, emission factors, reports)

Frontend: React.js + D3.js (interactive visualizations, Sankey diagrams)

Cloud Hosting: AWS (EC2, RDS, S3 for scalable deployment)

Pipeline: Apache Airflow (data ingestion, scheduling)

Data Sources:

Pipeline (Airflow Orchestration):

---

3. CORE CALCULATION ENGINE 🧠

The engine converts plant data into environmental metrics.

Steps:

1. Normalize Inputs → All units in common format.


2. Multiply with Emission Factors → Example:

Electricity: kWh × 0.82 kg CO₂/kWh

Diesel: Liters × 2.68 kg CO₂/liter

Anodes: kg × 3.0 kg CO₂/kg



3. Aggregate Emissions → Sum across mining, transport, refining, smelting, finishing.


4. Calculate Intensity → Total Emissions ÷ kg Aluminium produced.


5. Scenario Modeling → Users adjust inputs like renewable %, recycling %, or efficiency → engine recalculates instantly.



(Unique Add-On:) Incorporate water stress index (regional data) + waste-to-value analysis (e.g., red mud utilization).

DATA ACQUISITION (Step by Step)

1. Why Data Acquisition Matters

Life Cycle Assessment (LCA) is only as good as the data fed into it.

Aluminium plants already generate a lot of data (production logs, energy bills, ERP entries, transport invoices).

We need to collect, clean, and standardize this data into one system so the calculation engine can process it.



---

2. Types of Data We Need

We classify data into 4 main buckets:

a) Production Data

Tonnes of aluminium produced per month.

Tonnes of alumina processed.

Scrap aluminium percentage.


b) Energy Data

Electricity consumption (kWh) – smelting is electricity-heavy.

Fuel consumption (liters diesel, kg coal, kg natural gas).


c) Material Data

Carbon anodes used (kg).

Caustic soda, lime, water intake & recycled water (liters).


d) Transport & Waste Data

Distances travelled for raw material transport (tonne-km).

Waste generated (red mud tonnes, greenhouse gases, slag).



---

3. Sources of Data

Where does this data come from?

Internal Plant Data (Primary data):

ERP (Enterprise Resource Planning) systems.

SCADA (Supervisory Control & Data Acquisition) – real-time monitoring.

Meter readings (electricity, fuel tanks).

Purchase invoices (diesel, coal, anodes).

Production logs (how much aluminium was made).


External Data (Secondary data):

Grid emission factors (Central Electricity Authority – India).

IPCC or ecoinvent databases (global emission factors).

Carbon market price APIs (₹/tonne CO₂).




---

4. Data Acquisition Pipeline

We set up a two-layer collection strategy:

Layer 1 – Automated API Pull

For modern plants with ERP/SCADA → direct integration via API.

Example: Every midnight, Airflow fetches electricity data from plant ERP.


 Layer 2 – Manual Upload (Fallback)

For smaller plants or pilot testing → upload CSV/Excel templates.

Example: Plant manager uploads monthly diesel purchase invoice data.



---

5. Data Validation & Cleaning

Once data is acquired, we check:

1. Completeness → Are all fields filled? (e.g., missing electricity bill?)


2. Consistency → Units consistent (kg vs tonnes, kWh vs MWh).


3. Accuracy → Cross-check with invoices/logs.



(Unique Feature:) ML-based anomaly detection → if fuel use suddenly jumps 200% in one day, system flags it as error.


---

6. Normalization

Convert everything into standard units so the calculation engine can process.

Electricity → kWh

Fuels → MJ (megajoules)

Transport → tonne-km

Materials → kg



---

7. Storage

All validated & normalized data is stored in PostgreSQL database.

Metadata (who uploaded, timestamp, data source) is also logged for audit & compliance.


(USP): Store immutable copy on blockchain ledger → prevents data tampering when reporting to regulators.

ERP & SCADA in Aluminium Plants

1. ERP (Enterprise Resource Planning)

Think of ERP as the business brain of the factory.

It keeps track of orders, production logs, raw material purchases, energy bills, transport costs, and waste disposal invoices.


In aluminium plants, ERP (like SAP, Oracle, Microsoft Dynamics) stores:

Production Data → How much aluminium/alumina produced.

Materials Used → Carbon anodes, caustic soda, lime, etc.

Fuel Purchases → Diesel, coal, natural gas invoices.

Transport Manifests → Distance, weight, cost of moving bauxite/aluminium.


How to Fetch from ERP:

API Integration → Modern ERPs have APIs (like SAP OData API, REST API).

Example: GET /erp/production/monthly → returns tonnes of aluminium produced.


Database Connector → Direct read from ERP’s SQL tables.

Manual CSV Export → If API not available, managers export reports from ERP dashboard (CSV/Excel), then upload into our system.



---

2. SCADA (Supervisory Control & Data Acquisition)

SCADA is the real-time monitoring system for machines and utilities.

It collects data from sensors and meters in the plant → electricity usage, water flow, furnace temperatures, gas flow, etc.


 In aluminium plants, SCADA provides:

Electricity consumption (kWh) – smelting process meters.

Gas/coal usage (flow meters, weight sensors).

Water intake & recycling (flow sensors).

Process parameters (furnace efficiency, equipment uptime).


How to Fetch from SCADA:

OPC-UA / Modbus Protocols → Standard interfaces to extract data from SCADA.

SCADA Historian Database → Many SCADA systems (Siemens, ABB, Schneider) log historical data that can be queried via SQL or API.

Middleware Connectors → Tools like Ignition, Kepware act as bridges between SCADA and external systems.

CSV Exports → Small plants may export monthly reports.



---

3. How It Connects to Our LCA System

Automated API Pull (Preferred):

Airflow pipeline calls ERP & SCADA APIs at scheduled times (e.g., daily/weekly).

Data flows directly → reduces manual work, fewer errors.


Manual Upload (Fallback):

Managers export ERP/SCADA data in CSV/Excel.

Upload to our dashboard.

System validates & processes.
